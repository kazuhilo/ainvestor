# pr:new

**テンプレートを守ること**と**レビュアーに優しい本文を書くこと**を最優先する。

## レビュー依頼の心構え

これは PR 本文に書く規律であり、テンプレが先・心構えはそれを破らないための裏付け。

1. **レビュアーの時間は有限**。本文を「埋めるだけ」にしない。読み手が 30 秒で全体像をつかめるか自問する。
2. **What と Why は分ける**。What は事実、Why は動機。Why には「これをしないと何が困るか」を 1 文入れる。
3. **レビュー観点は優先順位つきで**。「特に見てほしい場所」と「見なくていい場所」を両方明示する（レビュアーが scope を絞れる）。
4. **検証済みの事実を書く**。実行ログ・テスト結果・自己レビュー観点を具体的に。未検証は「未検証」と明記し、黙って空欄にしない。
5. **期限は具体的に**。「お早めに」は禁止語。日付 or 「急ぎではない」を選ぶ。
6. **大きすぎる PR は分割提案**。diff が複数の独立判断を含むなら、PR 作成前にユーザーに分割を打診する（目安: 変更ファイル数 ≥ 15 or 1 PR で 2 つ以上の独立した意思決定を要求している）。
7. **タイトルは命令形・現在形で 1 行**。70 字以内。日本語可。例: `add pr:new` / `fix lifecycle 説明の誤記`。

## 実行手順

### 1. 前提取得

#### 1-1. リポジトリ状態の確認

並列で取得（依存なし）:
- `git rev-parse --show-toplevel` → `<ROOT>`
- `git branch --show-current` → `<BRANCH>`
- `git status --short` → uncommit 有無
- `git log --oneline origin/main..HEAD`（or `main..HEAD`）→ PR に含まれる commit 群
- `git diff --stat <base>...HEAD` → 変更ファイルサマリ

`<BRANCH>` が `main` / `master` の場合は中断し「PR 用の作業ブランチで実行してください」と案内。

#### 1-2. uncommit / unpushed の扱い

- uncommit がある場合: ユーザーに「先に commit しますか？／含めずに進めますか？」を AskUserQuestion で確認。勝手に `git add -A` しない。
- ブランチが remote に存在しない場合: PR 作成時に `gh pr create` 内で `--head` 指定 + 事前 `git push -u origin <BRANCH>` を行う旨を**ユーザーに告知してから**push する（CLAUDE.md「リスクある操作は確認」に従う）。

#### 1-3. テンプレ取得

`<ROOT>/.github/pull_request_template.md` を **Read** ツールで読み込む。
存在しなければユーザーに「PR テンプレートが見つかりません。作成しますか？」と問い、Yes なら作業を中断して手動作成を促す（このコマンドではテンプレ生成は行わない）。

テンプレは将来変更され得るため **このコマンドファイル内にコピーせず必ず読み込み時に取得**。

### 2. 下書き生成（diff / commit ログから自動推定）

以下を Claude が事前に下書きする。**ユーザー確認なしで PR は作らない**。

- **タイトル候補**: branch 名 + 最新 commit subject から 2〜3 候補を生成（70 字以内、命令形）。
- **What 下書き**: `git log --oneline <base>...HEAD` と `git diff --stat` から、追加 / 変更 / 削除 / 移動を箇条書きで列挙。
- **影響範囲**: 変更ファイル群をディレクトリ単位でグルーピング。

### 3. ヒアリング（テンプレの空欄を埋める）

テンプレの各セクションについて、下書きで埋まらない箇所を順に問う。
**埋まっているセクションも 1 度はユーザーに見せて確認**する（特に What 下書きの正確性）。

| セクション | ヒアリング方針 |
|---|---|
| What | 下書きを提示 → 「過不足ありますか？」と確認 |
| Why | **必ずユーザーに聞く**。下書き禁止。「この変更をしないと何が困りますか？」と踏み込んで質問する |
| レビューしてほしいポイント | Claude が候補を 2〜3 件提示（設計 / 規約 / UX 等から diff に該当する観点）→ ユーザーが優先度付け |
| 自分で確認したこと | セッション内で実行したテスト・動作確認をログから抽出して下書き → ユーザーに未検証項目を追加してもらう |
| 関連 | issue 番号・ADR パス・過去 PR をユーザーに確認（branch 名から推測できれば下書き） |
| レビュー希望期限 | AskUserQuestion で `YYYY-MM-DD` / `急ぎではない` / `Draft` の三択 + 自由入力 |
| マージ後のアクション | 下書きが思いつけば提示、なければ「特になし」を許容 |

3 ターン以上ヒアリングしても埋まらない項目は「未記入のまま投稿しますか？」と最終確認（黙って空欄にしない）。

### 4. プレビューと最終確認

組み立てた **PR タイトル + 本文** を Claude が 1 メッセージで全文プレビューし、ユーザーに最終承認を取る。
承認されるまで `gh pr create` を**絶対に呼ばない**。

「分割提案の閾値」に該当する場合（心構え 6）、プレビュー前に分割提案を一度行う。

### 5. PR 作成

1. 必要なら `git push -u origin <BRANCH>` を実行（事前告知済みの場合のみ）
2. 以下を実行（CLAUDE.md の HEREDOC 形式を踏襲、`Co-Authored-By` 行は PR 本文には付けない — commit message 専用）:

   ```bash
   gh pr create \
     --base <BASE_BRANCH> \
     --head <BRANCH> \
     --title "<タイトル>" \
     --body "$(cat <<'EOF'
   <本文>
   EOF
   )"
   ```

   Draft の場合は `--draft` を付与（ユーザーが「Draft」と答えた場合のみ）。
3. 戻り値の PR URL を取得。

### 6. 完了報告

ユーザーに 3〜5 行で報告:
- 作成した PR の URL
- タイトル
- ベースブランチ / ヘッドブランチ
- レビュー希望期限（再掲）
- 次のアクション案（例: 「レビュアー指定が必要なら `gh pr edit <num> --add-reviewer ...` で追加できます」）

## やってはいけないこと

- **テンプレートを使わずに PR を作る**（`.github/pull_request_template.md` の Read を省略しない）
- ユーザーの最終承認を取らずに `gh pr create` を実行する
- Why を自動推定で埋めて確認をスキップする（Why は必ずユーザーに聞く）
- 「お早めに」「なる早で」など曖昧な期限表記をテンプレに残す
- `--no-verify` 等で hook をスキップする（CLAUDE.md 違反）
- `git add -A` をユーザー確認なしに実行する
- 主要ブランチ（main / master）から直接 PR を作ろうとする
- 巨大 diff（≥ 15 ファイル or 複数の独立判断）の場合に分割提案をせず突き進む
- PR 本文に `Co-Authored-By:` 行を入れる（あれは commit message 専用）
- ユーザー指示なしにレビュアー / ラベル / マイルストーンを設定する
